# Сеточный торговый бот для Т-Инвестиций (Docker)

Готовый к развёртыванию сеточный (grid) торговый бот для [API Т-Инвестиций](https://developer.tbank.ru/invest/intro/intro).
Бот делит заданный ценовой диапазон на уровни, покупает начальную позицию и
выставляет стоп-заявки на покупку ниже цены и тейк-профиты на продажу выше:
на колебаниях цены внутри диапазона он покупает дешевле и продаёт дороже.

Проект упакован в Docker и рассчитан на запуск **в один клик** на
[Timeweb Cloud Apps](https://timeweb.cloud/my/apps): вы указываете свой токен
API в переменных окружения — и запускаете контейнер. Токен нигде не хранится,
кроме переменных окружения вашего приложения.

> ⚠️ **Дисклеймер.** Бот совершает сделки с **реальными деньгами** на вашем
> брокерском счёте. Это программное обеспечение предоставляется «как есть»,
> не является инвестиционной рекомендацией и не гарантирует прибыль.
> Вы используете его на свой страх и риск. Начните с небольшого капитала и
> убедитесь, что понимаете логику работы.

## Быстрый старт на Timeweb Cloud Apps

1. **Форкните** этот репозиторий на свой GitHub-аккаунт (кнопка Fork) —
   или используйте его напрямую, если у вас есть доступ.

2. **Выпустите токен** Т-Инвестиций: приложение/сайт Т-Инвестиций →
   Настройки → Токены API → создать токен **с полным доступом**
   (боту нужно выставлять заявки). Скопируйте токен — он показывается один раз.

3. В панели [timeweb.cloud/my/apps](https://timeweb.cloud/my/apps) нажмите
   **«Добавить App»** и выберите:
   - Тип: **Backend**;
   - Источник: ваш GitHub-репозиторий, ветка `main`;
   - Фреймворк / тип сборки: **Dockerfile** (файл `Dockerfile` в корне репозитория);
   - **Порт: `8000`**.

4. В разделе **«Переменные окружения»** добавьте:

   | Переменная | Пример | Обязательно | Описание |
   |---|---|---|---|
   | `TINKOFF_TOKEN` | `t.abc123...` | ✅ | Токен API Т-Инвестиций (полный доступ) |
   | `TICKER` | `SBER` | ✅ | Тикер инструмента |
   | `CLASS_CODE` | `TQBR` | — | Класс инструмента (по умолчанию `TQBR` — акции МосБиржи) |
   | `P_LOW` | `290` | ✅ | Нижняя граница диапазона, руб. |
   | `P_HIGH` | `330` | ✅ | Верхняя граница диапазона, руб. |
   | `CAPITAL` | `100000` | ✅ | Капитал бота, руб. |
   | `GRID_LEVELS` | `10` | ✅ | Количество уровней сетки |

5. Нажмите **«Запустить деплой»**. После сборки образа бот стартует
   автоматически: создаст сетку и начнёт торговать (или дождётся открытия
   биржи, если торги закрыты — это видно в логах приложения).

Все события бот пишет в лог — смотрите их прямо в панели Timeweb
(вкладка «Логи» приложения).

## Все переменные окружения

| Переменная | По умолчанию | Описание |
|---|---|---|
| `TINKOFF_TOKEN` | — | **Обязательно.** Токен API Т-Инвестиций. Также принимаются имена `T_INVEST_TOKEN`, `TBANK_TOKEN` |
| `TICKER` | — | Тикер инструмента (`SBER`, `GAZP`, ...) |
| `CLASS_CODE` | `TQBR` | Класс инструмента на бирже |
| `P_LOW` / `P_HIGH` | — | Границы ценового диапазона сетки, руб. |
| `CAPITAL` | — | Капитал, выделенный боту, руб. |
| `GRID_LEVELS` | — | Количество уровней сетки (`N`) |
| `ACCOUNT_ID` | первый открытый счёт | Идентификатор конкретного брокерского счёта |
| `AUTO_START` | `true` | Запускать бота автоматически при старте контейнера |
| `API_AUTH_TOKEN` | — (API отключено) | Ключ доступа к HTTP API управления (заголовок `X-API-Key`) |
| `ALLOW_EXISTING_POSITION` | `false` | Принять уже купленные лоты как начальную позицию бота |
| `RESET_ORDERS_ON_START` | `false` | Отменить при старте старые стоп-заявки по инструменту |
| `PORT` | `8000` | Порт HTTP-сервера |
| `LOG_LEVEL` | `INFO` | Уровень логирования (`DEBUG`/`INFO`/`WARNING`/`ERROR`) |

## Как работает стратегия

1. Диапазон `[P_LOW; P_HIGH]` делится на `GRID_LEVELS` уровней.
2. При запуске бот покупает по рынку начальную позицию примерно на половину
   капитала.
3. Ниже текущей цены выставляются стоп-заявки **на покупку**, выше —
   тейк-профиты **на продажу** (лоты и деньги распределяются по уровням).
4. Когда заявка исполняется, на парном уровне выставляется встречная:
   купил на уровне `i` → продаёт на `i+1`; продал на `i` → откупает на `i-1`.
5. Если цена выходит за границы диапазона, бот **не останавливается**:
   он ставит торговлю на паузу (уже выставленные заявки сохраняются),
   следит за ценой и разворачивает сетку заново при возврате в диапазон.
6. Если биржа закрыта, запуск откладывается до открытия торгов автоматически.

## Защита от двойной торговли

Контейнер на облачном хостинге эфемерен: при повторном деплое его локальная
база создаётся заново, а заявки и позиция на брокерском счёте — остаются.
Чтобы бот не купил начальную позицию второй раз и не выставил вторую сетку
поверх старой, при «чистом» старте выполняется проверка счёта:

- найдены **активные стоп-заявки** по инструменту, не относящиеся к текущему
  запуску → бот не стартует и объясняет это в логе. Решение: отменить заявки
  вручную в приложении Т-Инвестиций **или** задать `RESET_ORDERS_ON_START=true`
  (бот отменит их сам);
- найдена **позиция** по инструменту → бот не стартует. Решение: продать
  позицию вручную **или** задать `ALLOW_EXISTING_POSITION=true` — бот примет
  лоты как свою начальную позицию и продолжит без повторной покупки.

## HTTP API (опционально)

По умолчанию API управления **отключено** — бот работает автономно по
конфигурации из переменных окружения. Чтобы включить API, задайте
`API_AUTH_TOKEN` (длинная случайная строка, например `openssl rand -hex 32`)
и передавайте его в заголовке `X-API-Key`:

```bash
BASE=https://ваш-адрес-приложения
KEY="ваш API_AUTH_TOKEN"

# Список ботов
curl -H "X-API-Key: $KEY" "$BASE/api/grid-bot/list"

# Статус конкретного бота
curl -H "X-API-Key: $KEY" "$BASE/api/grid-bot/1"

# Остановить (снимает заявки бота) / запустить
curl -X POST -H "X-API-Key: $KEY" "$BASE/api/grid-bot/stop/1"
curl -X POST -H "X-API-Key: $KEY" "$BASE/api/grid-bot/start/1"

# Удалить бота и продать позицию по рынку
curl -X DELETE -H "X-API-Key: $KEY" "$BASE/api/grid-bot/1?sell_position=true"
```

Интерактивная документация: `GET /docs`. Проверка живости: `GET /health`
(без аутентификации).

## Локальный запуск

```bash
cp .env.example .env      # заполните TINKOFF_TOKEN и параметры сетки
docker compose up --build
```

Состояние (SQLite) при локальном запуске сохраняется в Docker-томе
`grid-bot-data`.

## Подключение к платформе deflow

Сервер можно подключить к платформе [deflow](https://deflow.ru) (раздел
«Боты на вашем сервере», `/app/trade/bots`) и управлять ботами и смотреть
статистику прямо на платформе. При этом:

- **токен Т-Инвестиций остаётся только у вас** — в переменных окружения
  вашего сервера. Платформа его никогда не запрашивает, не видит и не
  хранит: наружу отдаётся только булев флаг «токен задан/принят брокером»
  (`GET /api/platform/status` — публичный эндпоинт без секретов);
- платформа выполняет на вашем сервере только разрешённые операции:
  чтение данных (список ботов, статистика, сделки, график, позиция) и
  команды создать/запустить/остановить/удалить бота.

Как подключить:

1. Задайте на сервере переменную `API_AUTH_TOKEN` (длинная случайная
   строка, например `openssl rand -hex 32`) и перезапустите приложение.
2. На платформе deflow в разделе «Боты на вашем сервере» вставьте URL
   вашего приложения и этот ключ управления (не токен брокера!) и нажмите
   «Проверить и подключить».

Эндпоинты, добавленные для платформы:

| Метод | Путь | Аутентификация | Описание |
|---|---|---|---|
| GET | `/api/platform/status` | не требуется | Статус: `token_configured`, `token_valid` (`?verify=1`), `api_enabled`, `auth_ok`. Только булевы флаги, без секретов |
| GET | `/api/grid-bot/stats/{id}` | X-API-Key | Агрегированная статистика бота (прибыль, обороты, комиссии). `?sync=1` — подтянуть комиссии из операций брокера |
| GET | `/api/grid-bot/trades/{id}` | X-API-Key | История сделок (`?limit=&offset=`, `limit<=0` — все) |

## Безопасность

- **Токен только в переменных окружения.** Он не пишется в базу, файлы или
  логи. Никогда не коммитьте `.env` (он в `.gitignore`).
- **API закрыто по умолчанию.** Публичные эндпоинты без ключа — только
  `/health` и `/`. Управление ботом требует `X-API-Key`.
- **Непривилегированный контейнер.** Процесс работает от пользователя без
  прав root.
- **TLS с проверкой сертификатов.** Все запросы к брокеру идут по HTTPS
  с проверкой по CA bundle (certifi + Russian Trusted CA НУЦ Минцифры,
  которые скачиваются при сборке образа).
- Рекомендация: заведите **отдельный токен** для бота, чтобы его можно было
  отозвать независимо, и укажите конкретный счёт через `ACCOUNT_ID`.

## Важные ограничения

- **Ровно один экземпляр.** Не масштабируйте приложение на несколько
  реплик/воркеров — это приведёт к дублированию сделок.
- **Эфемерное хранилище.** История сделок в SQLite теряется при повторном
  деплое (заявки и позиция на счёте при этом сохраняются — см. раздел
  «Защита от двойной торговли»). Реальным источником правды всегда остаётся
  брокерский счёт.
- Один контейнер = один бот = один инструмент. Для другого инструмента
  создайте ещё одно приложение со своими переменными окружения.

## Лицензия и премиум-подписка

Бот торгует только при активной **премиум-подписке** на платформе deflow.
Схема построена так, чтобы:

- **платформа не имела доступа к вашему серверу.** Направление всегда одно:
  ваш сервер САМ раз в 15 минут обращается к платформе (исходящий HTTPS),
  отдаёт статистику и получает подписанный «lease». Платформа никогда не
  подключается к вашему серверу — поэтому её взлом не даёт доступа ни к серверу,
  ни к токену Т-Инвестиций (он лежит только в переменных окружения здесь);
- **скопированный код не работал.** lease подписан приватным ключом Ed25519,
  который есть только у платформы; в боте вшит лишь публичный ключ. Без
  валидных `LICENSE_ID`/`LICENSE_SECRET`, привязанных в БД платформы к
  премиум-пользователю, активный lease получить нельзя;
- **истечение подписки ставило бота на паузу.** lease живёт ~1 час (дольше
  15-минутного опроса), поэтому кратковременная недоступность платформы не
  роняет бота. Но как только подписка перестаёт быть премиальной, платформа
  прекращает продлевать lease, и бот в пределах часа встаёт на **паузу**:
  мониторинг продолжается, новые сетки не разворачиваются и новые заявки не
  выставляются, а **позиция и уже выставленные заявки не трогаются**.

### Настройка

1. В личном кабинете deflow нажмите «Выпустить лицензию» — получите
   `LICENSE_ID` и `LICENSE_SECRET` (секрет показывается один раз).
2. Задайте переменные окружения вашего сервера:

   | Переменная | Назначение |
   |---|---|
   | `LICENSE_ID` | идентификатор лицензии из кабинета |
   | `LICENSE_SECRET` | секрет лицензии (хранится только у вас) |
   | `DEFLOW_API_URL` | URL платформы, напр. `https://deflow.ru` |
   | `LICENSE_POLL_SECONDS` | интервал опроса, по умолчанию `900` (15 мин) |
   | `LICENSE_ENFORCE` | `true` (по умолчанию). `false` — только для локального теста |

> Модель угроз (честно): полностью запретить владельцу сервера вырезать
> проверку из исходного кода нельзя — сервер принадлежит ему. Схема
> останавливает случайных копировщиков и автоматически ставит бота на паузу
> при истечении подписки, не давая при этом платформе никакого доступа к
> вашим деньгам.

### Нативная компиляция движка (Cython)

Образ собирается в два этапа. На первом модули с торговой логикой и проверкой
лицензии компилируются в нативные `.so` (Cython), их исходники `.py` в образ
не попадают. Слой FastAPI/pydantic (роутеры, модели, настройки, точка входа)
остаётся исходником — иначе фреймворк не сможет читать сигнатуры и поля во
время работы. Список компилируемых модулей — в `build_native.py`.

Компиляция затрудняет реверс (нужно разбирать машинный код, а не байткод),
но это **деттерент, а не защита**: владелец машины при желании её обойдёт.
Основной барьер — лицензия и пауза без активной подписки.

Обычная сборка `docker build .` уже выполняет компиляцию. Отдельная команда
не нужна.
